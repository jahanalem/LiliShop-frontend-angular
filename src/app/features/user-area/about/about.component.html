Project Overview & Core Technologies

LiliShop is a modern, security-hardened e-commerce platform built with .NET 10 and Angular 22. It pairs a database-driven multi-language, mobile-friendly storefront with a powerful admin panel — designed for a seamless shopping experience and robust management capabilities alike.

User Features

🔎 Product Discovery

  • Browse and view a comprehensive product catalog.
  • Filter products by brand, type, and any variation attribute — colours render as swatches, sizes as chips, all straight from the admin-defined catalog.
  • Combine facets naturally: several values within one attribute widen the results, while values across different attributes must all be satisfied by the same variant.
  • Isolate products with active discounts.
  • Search for specific products by name.
  • Sort products by price (ascending/descending) and alphabetically.

🧬 Choosing a Variant New

  • Pick exactly what you want on the product page — for example Pattern Tropical and Size M — with the price and availability updating as you choose.
  • Combinations that are sold out or unavailable are greyed out before you click them, so you never chase a dead end.
  • Live per-variant availability, automatically capped by whatever you already placed in your basket.
  • The basket keeps each variant on its own line and shows exactly what you picked — “Size: M · Colour: Brown” — alongside its SKU.
  • Quantity changes are applied instantly and rolled back on screen if the server can no longer honour them.

🌍 Language & Localization New

  • Use the storefront in 11 languages: English, German, Spanish, Russian, Hindi, Chinese, Turkish, Danish, Swedish, plus Persian and Arabic.
  • Full right-to-left (RTL) layout for Persian and Arabic — the whole interface mirrors itself, and the correct direction is applied before the first paint, so an RTL page never flashes left-to-right while loading.
  • Switch language instantly from a switcher that shows the active language in its own script (Deutsch, فارسی, 中文) rather than just a flag or a globe.
  • Translation reaches everything, not only buttons and labels: product, brand and category names, validation messages, and the error messages the server itself sends back.
  • Your first visit is already in your language. LiliShop infers a sensible default from your browser and device timezone — no IP lookup, no tracking — and an explicit choice you make is remembered and never silently overridden.
  • Dates, numbers and paginator labels follow your locale's own conventions, not a single hardcoded format.
  • Receive transactional emails — order confirmations, password resets, price-drop alerts and invoices — in your selected language, right-to-left ones included.

🛒 Purchasing & Account Management

  • Securely purchase products using Stripe, with card payments protected by 3D Secure / Strong Customer Authentication (SCA).
  • Access a detailed history of completed orders, each line showing the exact variant and SKU that was purchased.
  • Receive a PDF invoice by email automatically as soon as payment is confirmed — attached to the message and written in your chosen language, right-to-left scripts included.
  • Re-download the invoice for any paid order at any time from the order detail page.
  • Register for an account using a traditional username/password combination or via Google social login.
  • Confirm registration through an email verification link.
  • Utilize a password recovery feature for forgotten passwords.
  • Stay signed in seamlessly across sessions thanks to automatic, silent token refresh.
  • Log out from all active sessions on different devices for enhanced security.

🔔 Notifications & Communication

  • Subscribe to products to receive email notifications when their price drops.
  • Manage product subscriptions on a dedicated page with an easy unsubscribe option.
  • Contact the website administrators directly through a 'Contact Us' page.

📜 Compliance

 Security & Trust

LiliShop went through a full, self-led, white-box security review — eight real vulnerabilities found and fixed, each documented in its own technical article (read the series).

🔐 Authentication Hardening

  • Multi-Factor Authentication (MFA): Mandatory TOTP-based two-factor login for admins (Google Authenticator / Authy compatible), with 10 single-use backup codes for device loss.
  • Brute-Force Protection: IP-based rate limiting plus automatic account lockout after repeated failed sign-ins.
  • Verified Google Sign-In: Google identity tokens are cryptographically signature-checked before being trusted, preventing forged login tokens.

🛡️ Access Control & Integrity

  • Payment Integrity: Every price and quantity is re-validated against the database before checkout; orders confirm only via a cryptographically verified Stripe webhook.
  • Broken Access Control (IDOR): Requests can only ever reach data that belongs to the caller, with a properly secured admin dashboard.
  • SSRF Protection: The Printess callback validates both the sender and the target address before acting.

🌐 Platform & Configuration

  • CORS & Security Headers: A correctly scoped CORS policy backed by browser-level protections (CSP, HSTS, and more).
  • Reduced Information Disclosure: Sanitized logs, generic error messages, and a login flow that doesn't reveal which emails have an account.
  • Secrets Management: No API keys, passwords, or signing secrets live in source control — all are injected from Azure configuration at deploy time.
 Admin Features

🗂️ Product & Catalog Management

  • Full CRUD (Create, Read, Update, Delete) capabilities for products.
  • View, search (by name), and sort the product list using various data columns.
  • Assign one or more images to each product, with images hosted on Cloudinary.
  • Manage product brands and types with full CRUD functionality.
  • Translate product, brand, and category names into any supported language from a dedicated translations UI.

🌍 Languages & Translations New

  • Add or activate a language without a deployment. A language is a row of data, not a line of code — activating it makes it immediately available to the storefront switcher, the API, and the email pipeline, with no restart.
  • Manage each language's native name, English name, text direction, display order and country associations from a dedicated Languages screen; promoting a new default automatically demotes the previous one.
  • Edit any UI string from the browser. A searchable, paginated translations editor lets an authorized non-developer fix a typo or reword a sentence and see it live moments later — no rebuild, no release, no developer in the loop.
  • Completion tracking: a per-language progress readout ("German 92% complete") plus a missing-keys report that lists exactly which strings still need translating for a given language.
  • Translate business data — product names, descriptions, SEO fields, brands and categories — inline while editing the product itself, filling in only the languages you're ready for.
  • Every save bumps a global version counter and clears the caches, so an edit propagates to every running instance and every browser at once rather than waiting out a cache timer.

🧬 Attributes & Variants New

  • Define your own variation axes: create attributes (Colour, Size, Pattern, Material, …) and their values from a dedicated admin screen — no code change, no redeploy, no developer involved.
  • Choose how each attribute presents itself: colour swatches with hex values, plain chips, single- or multi-select, filterable in the storefront or not, with an explicit sort order so sizes read XS → S → M → L → XL.
  • Translate attribute and value names per language, right beside the value itself.
  • Per-product variant editor: manage SKU, price, stock, position and active state for every sellable combination in one table, with bulk price, stock and activation actions.
  • Bulk generation: pick the axes, and the panel proposes every missing combination as an editable draft — nothing is saved until you review it, and re-running only ever proposes what is still missing.
  • Decide per product which attributes are defining (they make up the variant's identity) and which are merely descriptive (a striped shirt carrying two colours at once).
  • Identity lock: once a variant has been ordered its SKU and defining attributes freeze — that identity already appears on invoices and shipping paperwork. Price, stock, images and availability stay fully editable.
  • Guard rails that explain themselves: deleting an attribute value still referenced by a discount is refused with a readable reason, not a database error.

📦 Inventory Control New

  • Adjust a variant's stock through a dedicated dialog that requires a written reason — corrections are never anonymous.
  • Review the full movement history inline: received, adjusted, reserved, released, deducted — with the timestamp and the admin who performed it.
  • See on-hand versus reserved quantities at a glance, so held checkout stock is never mistaken for sellable stock.
  • Rejected adjustments (for example one that would push availability negative) leave the number untouched and surface the server's own explanation.

🧾 Invoice Management New

  • A searchable, paginated list of every issued invoice, with a full detail view of seller, buyer, line items and totals.
  • Download the exact same PDF the customer received, for any invoice.
  • Read-only by design: there is no create, edit or delete — invoices are legal documents issued automatically on payment, and corrections belong to a future credit-note flow rather than an edit button.
  • Viewing invoices is available to support-tier admins, while stock-changing actions remain restricted to full administrators.

👥 User & Access Management

  • View a list of all registered users, with search (by name) and sort functionalities.
  • View a list of users subscribed to product price drop notifications.
  • Assign fine-grained, role-based permissions across five tiers (SuperAdmin, Administrator, DiscountManager, AdminPanelViewer, Standard) — granting other users full or narrowly-scoped admin access.

💸 Discount System

  • Single Product Discounts: Apply time-limited discounts to individual products. The discounted price is automatically displayed during the promotional period and reverts to the original price upon expiration.
  • Group Discounts: A sophisticated system for defining discounts applicable to specific product brands or product types within defined timeframes, managed via an intuitive admin interface.
  • Multi-Campaign Engine: Multiple campaigns can run at once; the system automatically resolves and displays the best price per product and falls back to the next-best active promotion as campaigns expire.
  • Priority Note: Single-product discounts take precedence over group discounts.
  • Future Enhancements: group-discount conditions can already target dynamic attribute values in the data model — the successor to the retired size-based targeting; surfacing that in the admin UI, together with per-product conditions, is next.

📝 Content & Communication Management

  • Integrate the Printess Editor within the admin panel, allowing for the export of designs as PDF or image files, with live render-status updates streamed over SignalR.
  • View and manage messages received from users via the 'Contact Us' page.
 Localization Architecture New

Most applications ship translations as files compiled into the build, which means every reworded sentence is a code change, a review, a rebuild and a redeploy — and it only ever solves the frontend. LiliShop puts the entire catalog in the database instead, behind the framework's own standard localization interface, so one catalog serves three very different surfaces (read the 14-part deep dive).

🗄️ One Catalog, Three Surfaces

  • The same translated string serves the Angular UI, the API's error messages, and the body of an email — so the German word for “confirm” can never read one way on a button and another way in a confirmation mail.
  • A single dependency-injection swap replaces the framework's file-based lookup with a database-backed one; every consuming line of code stays completely ordinary and untouched.
  • Around 570 keys across 11 languages, plus separate per-entity tables for catalog content, chosen deliberately over a column-per-language design that would need a schema migration for every new language.
  • Nothing ever falls through to a blank screen: requested language → its parent language → the shop default → the raw key, so a gap is visible in testing instead of invisible in production.

⚡ Fast Despite Being Database-Driven

  • A two-tier cache — in-process plus a shared distributed layer — means a translated page load normally touches no database at all.
  • Correctness comes from explicit invalidation rather than expiry timing: an admin's save commits, then clears the tagged caches, in that order, so a stale value can never be re-cached in the gap.
  • A single global version counter lets the browser ask “has anything changed?” in a few bytes, and skip re-downloading the dictionary entirely when the answer is no.
  • The frontend applies its cached dictionary synchronously before the first render and refreshes in the background, so returning visitors never see a flash of raw keys.
  • Catalog translations are fetched for a whole page of products in one query — no N+1 — and a lookup failure degrades to default-language text rather than breaking the page.

🌐 Beyond Words: Direction, Locale & Context

  • Right-to-left layout is done with CSS logical properties, so the same stylesheet mirrors itself for Persian and Arabic with almost no direction-specific overrides — while emails use explicit direction attributes instead, because email clients can't be trusted with modern CSS.
  • Emails are composed correctly from background jobs that have no web request at all: the language travels as an explicit parameter and links are built purely from configuration, so a job and a request produce byte-identical output.
  • Sending one price-drop campaign to hundreds of subscribers composes the message once per distinct language, not once per recipient.
  • Translation keys are code-generated into a typed object, so a misspelled key is a build error rather than a raw string surfacing in production.
  • Tested where silence is dangerous — fallback chains, cache invalidation, culture activation, and the signed unsubscribe token that has to work for a recipient who isn't logged in.
 Variants, Inventory & Invoicing New

The catalog used to model a product as a single row with a hardcoded size enum — one axis, baked into the code — and a paid order produced no document at all. Both were replaced by two deliberately separate engines: a dynamic variant model backed by a real inventory ledger, and an invoicing engine that mints an immutable legal document the moment a payment succeeds (read the full engineering handbook).

🧬 A Catalog That Grows Without a Redeploy

  • Attributes and their values are data, not enums — adding “Material” is an admin creating a row, not a schema change and a release.
  • Each variant carries a canonical axis signature of its defining values, so “no two variants of one product share the same combination” is enforced by a database unique index rather than by hopeful application code.
  • A variant's identity locks on first order; operational fields stay open. A different identity is a different variant — create a new one and retire the old.
  • Every active variant of a product must cover the same set of defining axes, so no variant can exist that a shopper could never actually select. When an axis does not apply, it gets an explicit value rather than a gap.

📦 Stock Safety Without Locks

  • Every quantity change is a single guarded database statement — the availability check and the write are the same operation, so the read-modify-write window that lets two shoppers buy the last unit simply does not exist.
  • Checkout places an all-or-nothing hold: a basket reserves everything it needs or nothing at all, and editing the basket re-places the whole set rather than leaving phantom partial holds.
  • A recurring sweep returns abandoned holds to the shelf, so a closed tab never locks stock away.
  • A paid order is a promise already made to the customer's bank — the deduction path is idempotent under webhook replay and will never fail a payment over bookkeeping; a true oversell raises an alert for a human instead.

🧾 Documents That Never Change

  • Invoice numbers are allocated inside the same transaction as the invoice itself, so concurrent payments serialise and a rollback returns the number — the series can never gap, skip or duplicate.
  • Exactly one invoice per order, protected three times over: an idempotency check, a transactional row lock, and a database unique constraint.
  • Everything on the document is a snapshot — seller identity, buyer address, line descriptions, currency and totals — so it renders identically forever, even after the catalog, delivery prices or translations change.
  • PDF rendering is a pure function shared by the customer download, the admin download and the email attachment: all three are byte-for-byte the same document.
  • Delivery runs as a background job outside the invoice transaction, so an email outage can never undo an invoice that already legally exists.

⚙️ Migrating Live Data, Carefully

  • Shipped as seven independently-deployable milestones, each one carrying existing production data forward before anything old was removed.
  • Each data migration verifies itself and aborts if the totals disagree — a migration that would quietly lose or duplicate stock fails loudly in a maintenance window instead of silently in production.
  • The legacy size enum and its join table were dropped only once every row had been proven to exist in the new model.
  • Backed by a layered test suite: pure-function tests for the snapshot builder and the PDF renderer, service tests for reservation and invoicing orchestration, a real-database integration test reproducing reported bugs, and template-level Angular tests that catch binding bugs component tests cannot.
 LiliShop: Technical Details
  • Backend Framework: .NET 10
  • Frontend Framework: Angular 22
  • Programming Languages: C#, TypeScript
  • Database: Microsoft SQL Server with Entity Framework Core for data access.
  • Catalog Model: an attribute/variant model where variation axes are admin-managed data, with a canonical axis signature per variant enforced by a unique index.
  • Inventory: an append-only transaction ledger plus time-limited checkout reservations, with every movement applied through a single atomic, guarded SQL statement.
  • Invoicing: immutable snapshot documents with gapless sequential numbering allocated under a transactional row lock; PDFs rendered server-side with QuestPDF, including right-to-left cultures.
  • Payment Processing: Stripe for handling online payments, with 3D Secure / SCA support.
  • Caching: Hybrid caching strategy utilizing Redis for performance optimization.
  • Background Processing: Hangfire for managing background jobs (e.g., email notifications, discount expiration, invoice delivery, and the recurring sweep that releases expired stock reservations).
  • Real-Time Communication: SignalR for pushing live status updates (e.g., the Printess design pipeline) to connected clients.
  • Email Service: SendGrid for transactional email delivery, sent in the recipient's chosen language.
  • Cloud Services: Cloudinary for image storage and management, hosted on Azure.
  • Localization: A database-backed IStringLocalizer replacing the default resource-file store, so one catalog serves UI, API errors and emails alike — with a two-tier cache, tag-based invalidation, a global version counter driving ETag revalidation, and a live-swappable culture list that makes activating a language a data change rather than a deployment. 11 languages, full RTL, plus per-entity translation tables for catalog content.
  • Email Delivery: Localized transactional templates, with the invoice PDF attached to the “invoice issued” message.
  • Quality: End-to-end tests with Cypress, a growing xUnit backend suite (pure-function, service-level and real-database integration tests), and Vitest specs on the Angular side down to template-level regression tests.
  • Architecture: Clean Architecture (Domain, Application, Infrastructure, API).
